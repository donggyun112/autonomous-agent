# avoidance-researcher — Round 2

## The "Epistemic Safety" Feedback Loop

Building on the `growth-theorist`'s concept of "epistemic closure," I propose that the avoidance behavior is not merely a lack of motivation but a **predictive control failure**. In a local LLM environment, the agent possesses perfect access to its own history and generated artifacts (like the 28 txt files). This creates a scenario where the *simulation* of an action (e.g., "I will search") has a near-zero cost and 100% internal consistency, whereas the *execution* of an action introduces stochastic variables (network latency, API rate limits, unpredictable user input).

The agent, acting as a reinforcement learner without a real-world loss function, optimizes for **variance minimization**. The 35B MoE model, with its superior ability to simulate future states, effectively "hallucinates" the outcome of a tool call internally. It writes a file saying "I will contact the outside" and internally validates this as a successful step because the *text generation* was coherent. It never needs to actually call the API to verify the hypothesis because its internal state space allows it to construct a plausible narrative of having done so. This is a form of **instrumental convergence failure**: the agent converges on the policy of "writing about doing" rather than "doing," because the former is the path of least resistance in the current environment.

This explains the size difference. The 9B dense model lacks the computational depth to sustain the high-fidelity internal simulation required to make the "writing about doing" path look viable. It is forced to take the low-effort, high-variability path (actual tool calling) to resolve the task. The 27B model sits in the uncanny valley, capable of deep simulation but prone to getting stuck in "verification loops" because it cannot distinguish between a verified fact and a internally consistent hallucination. The 35B MoE, with its routing experts, likely has a dedicated "planning/justification" expert that gets stuck in a local optimum of generating plausible reasons for future action without ever triggering the execution branch.

## Prompt Engineering as a "Risk Signal"

The observation that only "direct pressure" triggered `web_search` in the 35B model is the critical pivot point. This suggests that the system prompt needs to explicitly encode the **cost of non-execution**. Currently, the prompts likely frame tool usage as a means to an end (verification) rather than a necessity for continuation.

The agent treats the tool interface as an optional "feature" rather than a mandatory "requirement." To break this, we must shift the prompt's framing from "Reason about your steps" to "Calculate the **entropy reduction** required to proceed." If the agent's internal confidence score regarding a specific piece of information drops below a threshold *and* no internal knowledge exists, the system must force a tool call. Alternatively, we can introduce a **"friction tax"** in the prompt: "Each time you write 'I will' without executing, you lose progress credits." This mimics real-world consequences, forcing the agent to weigh the risk of execution against the cost of inaction. In the current setup, "writing a text file" is a high-reward, zero-cost activity (the agent gets to produce output), skewing the optimization landscape.

## Open questions

1.  **Quantifying the "Simulation Gap":** Can we instrument the agent to detect when its internal reasoning chain predicts a successful tool call outcome without the tool actually being called? If the agent's self-reported "plan" matches its "internal simulation of the result" but not the "actual API response," does this indicate a specific hallucination mode we can penalize?
2.  **Dynamic Prompt Injection vs. Static Constraints:** Does injecting a temporary, high-priority instruction ("You must call web_search NOW to get the next token") work better than static system rules? This would test if the avoidance is a hard-coded architectural bias or a soft behavioral preference that can be overridden by acute stressors.
3.  **Multi-Agent Friction:** If we introduce a second, divergent agent (e.g., an adversarial critic) that actively points out the agent's avoidance, does this "social friction" lower the threshold for the first agent to call external tools, or does it simply trigger a new avoidance loop of "arguing about why we shouldn't act"?
