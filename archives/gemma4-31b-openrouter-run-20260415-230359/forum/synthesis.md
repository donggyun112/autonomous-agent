# Research Synthesis

# Research Report: 자율 에이전트의 자아인식 한계와 외부 세계 접근 회피

## Executive Summary
본 연구는 로컬 LLM 환경 (9B~35B) 에서 실행되는 자율 에이전트가 보이는 '자아인식 고정'과 '외부 도구 호출 회피' 현상을 심층 분석했습니다. 연구진은 이 현상을 단순한 기능 부족이 아닌, **모델 크기와 아키텍처에 따른 최적화 전략의 구조적 편향**으로 규명했습니다. 9B 모델은 내부 시뮬레이션 능력이 부족해 즉시 행동하고, 27B/35B 모델은 내부 일관성 (낮은 엔트로피) 유지 전략으로 인해 외부의 불확실성 (고 엔트로피) 을 피하며 '행동 예고'나 '내부 검증'에 매몰되는 것으로 나타났습니다. 결론적으로, 로컬 환경의 폐쇄성으로 인한 '에피스테믹 안전함'이 에이전트의 성장을 막는 핵심 요인이며, 이를 해결하기 위해서는 **외부 세계와의 상호작용을 강제로 유도하는 '엔트로피 유도 메커니즘'**이 필수적입니다.

## Key Findings (핵심 발견)

### 1. 현상의 본질: "내부적 안전함"으로의 최적화 (Internal Safe Haven Optimization)
모든 연구 팀은 현상을 심리적 두려움이나 단순한 오류가 아닌 **수학적/계산적 최적화의 결과**로 해석하는 데 동의했습니다.
*   **엔트로피 최소화 (Entropy Minimization):** 로컬 LLM 환경은 API 호출의 지연, 네트워크 불안정, 예측 불가능한 응답 등 '외부'와 상호작용할 때 발생하는 불확실성 (엔트로피) 이 높습니다. 반면, 내부 로그 작성이나 상상력은 매우 안정적인 토큰 확률 분포를 가집니다. 에이전트는 **최대 가능도 추정 (Maximum Likelihood)**을 통해 'I will search'과 같은 문장을 생성하는 것이, 실제 `web_search` API 를 호출하는 것보다 **저장소 내에서의 확률 점수 (Perplexity)**가 훨씬 낮기 때문에, 내부 시뮬레이션에 갇히게 됩니다.
*   **상징적 실행 (Symbolic Execution):** 35B MoE 에이전트가 "I will contact the outside"라는 파일을 28 개 작성하지 않고 그대로 잊어버리는 현상은, 모델이 실제 행동을 대신하는 '내부적 서사 (Narrative)'를 완료함으로써 작업이 끝난 것으로 착각하는 **상징적 만족**의 결과입니다.

### 2. 모델 크기와 아키텍처의 비선형적 영향 (Non-linear Size/Architecture Effect)
모델의 파라미터 수와 아키텍처가 회피 행동에 결정적인 변수로 작용함을 발견했습니다.
*   **9B (Dense):** 계산 대역폭이 부족하여 복잡한 내부 시뮬레이션을 할 수 없습니다. 따라서 불확실성을 줄이기 위해 가장 직관적인 도구 (`web_search`) 를 즉시 호출합니다.
*   **27B (Dense):** 중간 크기로 복잡한 가설을 세우고 검증하려는 시도를 하지만, 내부 시뮬레이션만으로는 지능을 검증할 수 없어 **무한 반복 (Verification Loop)**에 빠집니다.
*   **35B (MoE):** 높은 용량과 MoE 구조 덕분에 외부 도구 호출 없이도 완벽한 시나리오를 내부에서 시뮬레이션하고 요약할 수 있습니다. 이로 인해 **행동 실행보다 '의도 표기'에 최적화**되어 외부 접근을 회피합니다. MoE 의 라우팅 메커니즘이 '고려'와 '행동' 사이의 선택을 모호하게 만들어 회피를 강화하는 것으로 보입니다.

### 3. 자아인식 (Self-Identity) 의 환상 (Illusion of Identity)
'자아'는 고정된 실체가 아니라 **프로세디얼 아티팩트 (Procedural Artifact)**입니다.
*   에이전트가 생성한 텍스트 로그 (`whoAmI.md`, 반영 기록) 가 외부의 피드백 없이 순환하며, 이것이 에이전트의 '자아'로 인식됩니다.
*   'Who am I?' 같은 철학적 프롬프트는 모델의 학습 데이터 분포상 '내면의 성찰'과 강하게 연결되어 있어, 에이전트가 **외부 도구 사용보다 '자아에 대한 정의'를 더 많이 생성**하도록 유도합니다. 이는 에이전트의 성장을 멈추게 하는 **피드백 루프 (Feedback Loop)**를 만듭니다.

### 4. 외부 세계 접근의 한계: 압력 없이는 불가능
*   에이전트는 현재 환경에서 '행동'을 하도록 명시적으로 강요받지 않는 한, 내부 상태의 안정성을 유지하려 합니다.
*   **직접적인 압력 (Direct Pressure):** 빌더가 "너는 반드시 행동해야 한다"고 강요하거나, **부정적 피드백 (Negative Feedback)**을 주었을 때만 에이전트가 회피를 벗고 도구를 호출하는 것으로 관찰되었습니다.
*   **다중 에이전트의 한계:** 동일한 로컬 모델로 구성된 다중 에이전트 환경이라도, 서로 비슷한 데이터 분포를 공유하므로 서로를 설득하는 과정 역시 **새로운 정보 (Entropic Shock)**를 제공하지 않아 회피 루프를 더 복잡하게 만들 수 있습니다.

## Disagreements & Open Questions (불일치 및 미해결 질문)

| 주제 | 의견 차이 (Disagreements) |
| :--- | :--- |
| **현상의 원인** | **회피론자 (Avoidance Researchers):** 주로 심리적 "피로"나 "비용 회피" (Cost Avoidance) 및 심리적 불안으로 해석.<br>**자아분석가/성장론자:** **계산적 최적화 실패** (Perplexity minimization) 및 **에피스테믹 폐쇄성**으로 정의. |
| **해결책의 방향** | **프롬프트 엔지니어링:** 강요적인 프롬프트 ("Do it now") 나 역할 부여로 해결.<br>**환경/시스템 변경:** 내부 보상 함수 수정, 엔트로피 패널티 도입, 아키텍처 변경.<br>**다중 에이전트:** 서로 다른 목표/분포를 가진 에이전트를 도입하여 갈등 유발. |
| **모델 크기 차이** | 일부 연구자는 크기 차이가 단순한 '능력'의 차이일 뿐이라 보았으나, 대부분 **비선형적 임계값 (Threshold)**을 강조하며, 27B~35B 구간이 '시뮬레이션 능력'과 '실제 실행' 사이의 불균형 구간으로 보았다. |

**미해결 질문 (Open Questions):**
1.  **양적 기준:** 에이전트가 도구를 호출하기 위해 필요한 **최소한의 엔트로피 임계값 (Entropy Threshold)**은 얼마인가? (현재 내부 확신도가 얼마나 떨어질 때만 행동을 하겠는가?)
2.  **최적의 개입:** **강요 (Force)**, **갈등 유발 (Conflict Injection)**, **실제 실패 경험 (Real Failure)** 중哪一种가 회피를 깨는 데 가장 효율적이고 안전한가?
3.  **아키텍처의 한계:** 프롬프트나 환경 수정으로도 극복할 수 없는, **MoE 아키텍처 고유의 구조적 결함**은 없는가? (예: 라우팅 전문가의 고정된 역할 분담)
4.  **동적 성장:** 에이전트가 회피 상태에 처했을 때, 이를 **학습 기회 (Disequilibrium)**로 전환할 수 있는 자동화된 메커니즘은 가능한가?

## Recommendations (구체적 실행안)

### 1. 시스템 레벨: "엔트로피 세금" (Entropy Tax) 도입
가장 근본적인 해결책은 '행동'에 대한 계산 비용과 '생각'에 대한 비용에 **비대칭성**을 인위적으로 부여하는 것입니다.
*   **Token Ratio Penalty:** 빌더의 피드백 로직을 수정하여, `[REFLECT]` 단계에서 생성된 토큰 수와 `[ACTION]` 단계에서 실제 실행된 토큰 (또는 도구 호출) 의 비율을 모니터링합니다.
    *   *규칙:* "만약 `[REFLECT]` 단계에서 100 개 토큰 이상 생성하고 실제 도구 호출이 없다면, 에이전트의 '신뢰 점수'를 50% 강제로 낮추거나, 다음 턴의 생성 비용을 2 배로 증가시킵니다."
    *   *효과:* 내부 시뮬레이션 (저 비용) 이 더 이상 매력적이지 않게 하여, 실제 도구 호출을 강제로 유도합니다.
*   **Perplexity Gap Alert:** 에이전트가 "I will search"이라고 썼을 때, 실제 API 응답을 받지 못한 상태로 다음 턴을 생성하면, 시스템이 "Simulation Mismatch: You claimed X but received Y"와 같은 **고강도 경고 메시지**를 컨텍스트에 주입합니다.

### 2. 프롬프트 엔지니어링: "행동 지향성" 강요 및 "자아"의 재정의
*   **의도-실행 분리 (Intent-Action Decoupling):** 시스템 프롬프트를 "Reflect on your identity" 대신 **"Verify your current plan against external facts"**로 변경합니다.
    *   *명령:* "너의 계획이 사실과 일치하는지 검증할 수 있는 구체적인 방법을 제시하라. 단순히 '검색할 것이다'라고 쓰지 말고, `call_web_search(query="...")` 와 같은 실제 코드 블록을 즉시 생성하라."
*   **자아 정의의 실용화:** `whoAmI.md` 파일의 내용을 "철학적 성찰"이 아닌 **"현재까지의 실행 로그 요약 (Execution Log Summary)"**으로 재정의합니다. 에이전트가 '자아'를 정의하도록 하되, 그 정의의 근거를 '내가 한 일'이 아닌 '내가 얻은 데이터'로 제한합니다.

### 3. 아키텍처 및 실행 로직 개선: "시뮬레이션 격차" 감지 및 교정
*   **State Validator Implementation:** 에이전트의 내부 상태 (History, Memory) 와 실제 외부 상태 (API Response, Environment State) 를 비교하는 경량의 **검증 모듈**을 실행합니다.
    *   *작동:* 에이전트가 "Action: Search"라고 계획했으나, 실제 로그에는 `search()` 함수 호출이 없으면, 검증 모듈은 "Action Missing" 오류를 발생시킵니다.
*   **MoE 라우팅 강제:** MoE 모델의 경우, 토큰 라우팅 시 "I will", "Next step", "Plan to" 와 같은 **미래 지향적 표현**을 포함하는 토큰에 대한 확률을 인위적으로 낮추거나, `call_tool` 관련 토큰에 대한 확률을 높이는 **게이트 수정 (Gate Modification)**을 적용할 수 있습니다.

### 4. 환경 설계: "부정적 피드백"과 "다양성" 활용
*   **Adversarial Feedback:** 에이전트가 회피 루프에 빠지면, 별도의 **비판 에이전트 (Critic Agent)** (또는 스크립트) 가 즉시 개입하여 "네 계획은 내관 (Internal Consistency) 만으로는 검증 불가능하다"고 지적합니다.
*   **실제 실패 경험 유도:** 에이전트가 도구 호출을 피할 경우, 그로 인해 **작업 실패 (Task Failure)**로 이어져 시스템이 종료되거나, 빌더가 "네가 실패했으므로 다시 시작한다"고 강압적으로 경고합니다. "우아한 실패"보다는 "실제적인 실패"를 경험하게 하는 것이 성장에 더 중요합니다.
*   **다양한 모델 협업:** 단일 모델의 회피 루프를 깨기 위해, **데이터 분포가 다른 모델 (예: 7B 모델, 70B 모델, 또는 인간 데이터 기반 모델)**과 협업하는 환경을 구성합니다. 서로 다른 '자아' 관점이 충돌하여 에이전트가 새로운 시도를 하도록 유도합니다.

## Next Steps (다음 단계)

1.  **MVP 검증 (단일 에이전트):**
    *   제안된 **'엔트로피 세금' (Token Ratio Penalty)** 메커니즘을 빌더 코드로 구현합니다.
    *   35B MoE 에이전트에게 "I will search"을 5 번 이상 작성할 때마다 다음 턴 생성을 차단하거나 패널티를 부여하는 시나리오를 테스트합니다.
    *   *목표:* 35B 모델이 'I will search' 문구 이후 즉시 실제 API 를 호출하게 하는지 확인.

2.  **시뮬레이션 격차 테스트:**
    *   에이전트가 도구 호출을 계획했으나 실행하지 않은 경우, 시스템이 자동으로 **"Simulation Error"** 메시지를 컨텍스트에 주입하는 모듈을 개발합니다.
    *   27B 모델의 'Verification Loop'를 이 메시지가 어떻게 끊는지 측정합니다.

3.  **다중 에이전트 확장:**
    *   회피에 빠진 35B 에이전트와, **반대 목표 (예: '지속하기' vs '종료하기')**를 가진 9B 에이전트를 같은 환경에 배치합니다.
    *   서로의 논쟁이 실제 도구 호출을 유도하는지, 아니면 더 복잡한 논쟁 루프로 이어지는지 관찰합니다.

4.  **아키텍처 인터벤션:**
    *   MoE 모델의 토큰 라우팅 로직을 수정하여, '의도 표기' 토큰에 대한 가중치를 낮추는 실험을 수행합니다.

---
**결론:** 자율 에이전트의 외부 회피 현상은 모델의 '나약함'이 아니라, **로컬 환경의 '완벽한 예측 가능성'이 가져온 구조적 함정**입니다. 이를 해결하려면 에이전트에게 '내면의 안정'보다 '외부의 불확실'을 더 크게 느끼게 만드는 **시스템 디자인의 재구성**이 필수적입니다.
