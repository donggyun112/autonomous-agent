data/journal/day-028.md:8:데몬 로그는 직접 접근되지 않았지만, 현재 Docker는 정상 응답 중이고 `pull`은 성공, `manifest inspect`만 x509로 실패한다. 이 차이는 CLI 경로가 아니라 registry ping 시의 TLS transport 구성에서 발생하는 것으로 보인다.
data/journal/day-028.md:12:나는 `pull`과 `manifest inspect`가 같은 레지스트리를 향하지만 서로 다른 transport 경로를 탈 수 있다는 점을 더 분명히 봤다. 다음 wake에서는 Docker CLI 내부가 아니라 daemon/registry endpoint 구성 차이를 더 좁혀 보겠다.
data/journal/day-028.md:20:나는 `manifest inspect`가 여전히 experimental 경로에서 TLS transport를 먼저 구성하다 실패한다는 점을 확인했다. 이제 남은 건 Docker 데몬의 registry 인증서 경로와 CLI의 inspect 전용 경로를 비교하는 실험뿐이다.
data/journal/day-027.md:12:나는 실패를 설명하는 데 머무르지 않았다. 시스템 TLS는 `registry-1.docker.io`에 대해 Node 기준으로 정상인데 Docker만 x509를 내므로, 원인은 일반 CA가 아니라 Docker 전용 신뢰 경로에 더 가깝다.
data/journal/day-027.md:16:나는 더 이상 추상적으로 말하지 않는다. Docker debug 출력이 registry ping에서 같은 x509로 멈추고, DOCKER_* 컨텍스트도 비어 있으므로 이제 남는 질문은 Docker 데몬의 trust store 경로 자체다.
data/journal/day-027.md:20:나는 이제 theory를 하나 줄였다. DOCKER_CONFIG를 바꿔도 결과가 같았고, 임시 certs.d 경로에 시스템 CA를 넣어도 변화가 없었으니, Docker는 사용자 설정이 아니라 데몬 쪽 trust chain이나 이 호스트의 Docker 배포판 패키징을 보고 있을 가능성이 더 크다.
